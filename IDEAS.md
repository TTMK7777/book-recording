# 読書履歴管理ツール アイデアメモ

作成: 2026-04-21
ステータス: アイデア出し段階（優先度低）
目的: 読書履歴をもっと有効に管理して読書モチベを高める

## 技術スタック方針（2026-04-21 決定）

**Next.js + shadcn/ui + Vercel** を採用。

理由:
- 技術的な知見を広げることが本プロジェクトの副次目的
- TSスタック保有のため言語学習コスト不要
- 外部公開・SNSシェア（LinkedIn発信との連携）を見据えると Streamlit では天井に当たる
- Vercelデプロイで共有URL即発行、OGP自動生成、モバイル対応が無料で手に入る

Streamlitを採用しない理由:
- URL設計不可（`/books/[isbn]` のような個別ページが作れない）
- OGP画像生成不可（LinkedInシェアで絵が出ない）
- デザイン自由度の天井が低い
- 詳細比較は本README内「Streamlitの天井」節参照（→ 別途 docs/ 化予定）

## 1. 先行AI分析（出発点）

### 1.1 Kindle公式APIの現状
Amazon/Kindleは読書履歴・ハイライト・メモを公式APIとして外部開発者に開放していない。

- **KDP API**: 著者・出版社向け、読者データ不可
- **Product Advertising API**: 書誌メタデータのみ、個人の読書履歴との紐付け不可
- **個人の読書履歴・ハイライト取得**: 公式手段なし

→ 「Kindleの連続読書記録をAPIで引っ張る」ことは**できない**。

### 1.2 実現ルート（3系統）

#### ルートA: 既存サービス（Readwise等）基盤
- **仕組み**: Readwiseが `read.amazon.com/notebook` をスクレイピング取得、APIで外部提供
- **メリット**: ハイライト取得即解決、Readwise APIは公開
- **デメリット**: 月額 ~$8/月、Amazon規約的にグレー、サービス依存
- **取得可**: ハイライト、メモ、書籍タイトル、著者
- **取得不可**: 読書時間、連続記録、ページ進捗

#### ルートB: 自前エクスポート + 取り込み
- **仕組み**: Kindleアプリの「メモとハイライトをエクスポート」→メール→取り込み、または `My Clippings.txt` パース
- **メリット**: 無料、規約クリーン、データ主権
- **デメリット**: 手動トリガー、書籍ごとに操作、自動化しにくい
- **取得可**: ハイライト、メモ、位置情報
- **取得不可**: 読書活動データ

#### ルートC: 画面キャプチャ + OCR/AI解析
- スクショをClaude APIに投げてメタデータ構造化
- 連続記録・進捗は取れるが**運用負荷高く現実性低い**

### 1.3 レイヤ別の取得元マトリクス

| レイヤ | 必要データ | 取得元 |
|---|---|---|
| 書誌メタデータ | タイトル、著者、ページ数、ジャンル | Amazon PA-API / Google Books API |
| ハイライト/メモ | 抜粋テキスト、位置 | Readwise API / 自前エクスポート |
| 読書活動（連続日数等） | 日別読書フラグ | 取得不可→手動 or 諦め |
| 統合・検索UI | - | Claude Code自作 |

### 1.4 先行AIの推奨MVP
1. Google Books APIで書誌メタデータ取得
2. Kindleメモ/ハイライトをメールエクスポート→Gmail API連携で自動取り込み
3. ローカルSQLite + Claude Codeで検索・タグ付け・要約生成
4. 読書活動記録はKindleアプリ任せにして自分のアプリでは扱わない割り切り

**追加課金ゼロ・規約クリーン・2〜3日でプロトタイプ**。

### 1.5 先行AIが挙げたリスク
- Readwiseのスクレイピング依存（Amazon仕様変更で即停止リスク）
- 「作ること」自体が目的化する罠（既存ツールで80点）
- 時間配分（診断士試験8月 + LinkedIn発信戦略との両立）

## 2. 先行分析で抜けている/弱い視点（追加検討）

### A. データソースの見落とし（日本人ユーザー視点）

先行分析はKindleに寄りすぎ。日本語書籍対応の選択肢が抜けている。

| ソース | 取得可否 | 備考 |
|---|---|---|
| **読書メーター** | スクレイピング可（非公式） | 日本最大の読書SNS。読了日・感想・★あり |
| **ブクログ** | RSS/HTML取得可 | 本棚公開URLあり、API相当 |
| **カーリルAPI** | 公式・無料 | 全国図書館の蔵書・貸出、ISBN正規化に最強 |
| **国会図書館サーチAPI** | 公式・無料 | 書誌メタデータの権威 |
| **openBD** | 公式・無料・認証不要 | 日本書籍特化、和書カバレッジ最強 |
| **Audible** | Amazon系だが非公式library取得手段あり | 「聴く読書」も履歴化 |
| **物理本（ISBNバーコード）** | スマホカメラ + quagga.js等 | 物理本取り込みが完全に抜けている |

**→ Google Books API単体では和書カバレッジが弱い。openBDが強力。**

### B. 「読書活動データ取得不可」は諦めすぎ

代替案が複数ある：

- **iOSスクリーンタイム**（ScreenTime API経由でKindleアプリ使用時間取得）
- **iOSショートカット+オートメーション**でKindle起動時に日付自動記録
- **Kindle for PCのローカルDB**（`My Kindle Content` 配下、進捗情報含む）
- **手動5秒記録UI**（Slackボット/LINE bot/iOSウィジェット）
- **端末のアプリ使用統計**を日次バッチで取得

### C. モチベーション設計の視点が完全に欠落

「読書管理」寄りで、本来の目的（モチベを高める）が放置されている。

- **ゲーミフィケーション**: ストリーク、ジャンルカバレッジレーダー、月次MVP本
- **振り返り自動生成**: 四半期「読んだ本から見るあなたの思考トレンド」AI生成
- **LinkedIn発信連携**: ハイライト→投稿下書きをClaude APIで自動生成 ← 発信戦略とシナジー大
- **「次に読む本」AI推薦**: 過去ハイライトを埋め込みベクトル化→類似本提案

### D. アーキテクチャ選択肢の不足

先行分析は「Claude Code自作 + SQLite」一択。既存資産活用の道も：

- **Obsidianプラグイン化**（Obsidian使っていればDataviewで80点）
- **MCPサーバー化**（Claude Desktopから自然言語で本棚操作、`mcp-book`的サーバー自作）
- **Notion DB + Claude API**（UIゼロで書けるのが速い）
- **ローカルSvelteKit + SQLite**（単一バイナリ化可、Claude Code的最速）

### E. Build vs Buy の検討が表層的

- 既存ツールで何%解決するか（Readwise Reader + Notion AIで体感90%）
- 自作の独自価値は1文で言えるか
- 自作するなら最大の差別化は **「日本書籍対応」+「試験/発信との連携」**

## 4. 現時点の推薦仮説

**最小構成**: Obsidianプラグイン or MCPサーバー + openBD/カーリルAPI + Kindleメール取り込み + **LinkedIn投稿下書き生成を目玉機能に**。

**独自価値の2点セット**:
- 和書強い（openBD/カーリル/国会図書館）
- ハイライト→SNS発信の自動ブリッジ

これならReadwise系では代替不可。
